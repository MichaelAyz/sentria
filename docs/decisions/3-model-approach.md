# ADR 3: Model Approach (Classic ML with Offline Training & Boot Loading)

## Context
Sentria needs to classify unstructured log messages into categories (`error`, `warning`, `security`, `noise`) and output a confidence score. While deep learning models (such as HuggingFace Transformers / BERT) offer high accuracy and understand deep semantics, they present severe engineering drawbacks for this use case:
1. **Inference Latency**: Transformers on CPU can easily take 50ms to 200ms per classification, causing bottlenecks in high-volume log pipelines.
2. **Resource Consumption**: Deep learning frameworks require heavy memory allocations and PyTorch runtime.
3. **Container Size**: A transformer-based container easily balloons to 2GB–4GB, making cold starts slow and deployment expensive.
4. **Task Domain**: Log files are short, structured, and use a limited vocabulary. Deep semantic understanding of natural language is rarely needed.

## Decision
We will use a **Classic Machine Learning Pipeline**:
- **Feature Extraction**: TF-IDF (Term Frequency-Inverse Document Frequency) Vectorizer.
- **Classifier**: `SGDClassifier` (configured with `loss='log_loss'` to output calibrated probabilities for confidence scores) from `scikit-learn`.
  - *Note on Calibration*: Linear models with weak regularization push probabilities to extremes even near the decision boundary. We explicitly set the `alpha` parameter (L2 regularization) higher than the default (e.g. `0.0005` vs `0.0001`) to provide smoother, better-calibrated probability estimates. This ensures that genuinely ambiguous logs correctly reflect a lower confidence score instead of being artificially inflated to near 1.0.
- **Training Story**:
  - Training occurs **offline** using a standalone script `training/train.py`.
  - It runs against a labeled dataset generated by the synthetic generator in batch mode.
  - The script trains the pipeline (TF-IDF + Classifier) and serializes it to `service/model.joblib` using `joblib`.
- **Inference Lifecycle**:
  - The FastAPI application loads `model.joblib` **exactly once** during the startup event (using FastAPI's `lifespan` handler) and stores it in memory.
  - The service *never* fits, trains, or recalculates vocabulary during incoming HTTP requests.
- **Packaging/Runtime Weight Strategy**:
  - The serialized model file (`model.joblib`) is **baked directly into the Docker image**.
  - Tradeoff: Baking the model file (~2-5MB) into the image makes the image self-contained and guarantees that the container has everything it needs to run, but increases image size slightly and requires a rebuild/re-deploy to update the model. Mounting as a volume keeps the image generic and allows updating the model dynamically, but increases operational complexity (volume provisioners, EKS mounts, or init containers).
- **Model Versioning Failure Mode**:
  - The git SHA is expected to be passed via the `--build-arg GIT_SHA` in the Dockerfile, rather than relying on the `.git` context being available inside the container. 
  - If the SHA cannot be determined at build or runtime (e.g. ad-hoc local testing), the service will fallback to reporting `"unknown"` instead of crashing. This is treated as degraded observability, not a fatal functional error, which contrasts with the missing model weights (which will cleanly crash the process).

## Consequences
- **Sub-10ms Latency**: Classic ML pipelines process text inputs in single-digit milliseconds on CPU, satisfying high-throughput requirements.
- **Minimal Image Size**: The image needs only `scikit-learn`, `joblib`, and `FastAPI`, keeping the final Docker image under 200MB (instead of 2GB+).
- **Self-Contained Deployments**: Baking the model weights directly into the Docker image ensures that the container is fully self-contained and reproducible. Updating the model requires a normal CI/CD rollout, which matches standard software release gates.
- **Cold-Start Correctness**: Because the model is loaded once at boot time, the first `/classify` request is as fast as subsequent requests, avoiding typical cold-start performance degradation in ML endpoints.
- **Simplicity**: No external model registry (like MLflow) or S3 object store is needed to pull weights at boot time, minimizing moving parts.
